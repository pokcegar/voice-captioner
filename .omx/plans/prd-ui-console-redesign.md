# PRD — VoiceCaptioner 中控台式 UI 重构

日期：2026-05-15  
来源：`.omx/specs/deep-interview-ui-console-redesign.md`  
状态：已规划，待执行

## 1. 背景与问题

当前 VoiceCaptioner 已具备本地录音、延迟实时草稿、本地 Whisper 转写、Markdown/SRT/JSON 导出和历史列表。但详情区仍是单列滚动堆叠：历史列表之外，录音设置、模型设置、录音按钮、转写按钮、权限、预览和会议详情在同一纵向工作流中出现（`Sources/VoiceCaptionerApp/ContentView.swift:16-35`）。

用户反馈的核心问题不是功能缺少，而是**设置和使用入口混在一起**：

- `语言` picker 位于 `录音` section（`ContentView.swift:75-92`），看起来像“会议语音语言”，但实际是 UI 语言。
- 会议中交流/转写语言的选择尚未做；第一版已决策为**固定中文转写**，不得出现让用户误会的语言选择入口。
- 中间当前会议工作区不突出；录音状态、延迟实时字幕、最终转写预览/修改没有形成主任务流。
- 最终 Markdown 需要允许用户轻量修改，但机器原始 `final.md` 必须保留不动。

## 2. 产品目标

把 VoiceCaptioner v1 的桌面端改成“会议中控台”：

1. 左侧：会议历史。
2. 中间：当前会议主工作区，承载录音状态、主要操作、字幕显示、最终转写预览和轻量 Markdown 编辑。
3. 右侧：设置检查器，按“常用优先 + 高级折叠”组织，并在录音中弱化/收起。
4. 明确区分：
   - **界面语言**：中文/English/Deutsch，仅影响 UI 文案。
   - **转写语言**：第一版固定中文，只显示只读说明，不做 picker。
5. 用户编辑后的 Markdown 另存为独立文件，不覆盖机器生成的 `final.md`。

## 3. 非目标

第一版不做：

- 多会议语音语言选择。
- 自动语言识别。
- 翻译/双语输出。
- 富 Markdown 编辑器或 WYSIWYG。
- 分段字幕编辑器。
- 将用户编辑 Markdown 反向同步到 SRT/JSON。
- 真零延迟实时字幕承诺；继续沿用“延迟实时草稿”。

## 4. 用户故事

### US1：开会前快速开始

作为会议记录用户，我打开 App 后能在左侧看到历史，中间看到当前/选中会议状态和主要录音按钮，右侧只在需要时调整输出目录、标题、模型等设置，这样不会把配置误认为会议内容。

### US2：录音中看字幕

作为正在开会的用户，我录音时主要关注中间区域：录音状态、停止按钮、延迟实时转写草稿。右侧设置自动弱化，避免干扰。

### US3：录音后修正文稿

作为用户，我完成本地转写后可以在中间用简单纯文本 Markdown 编辑器修正文稿；保存后生成用户编辑版 Markdown，机器原始 `final.md` 保持不变，方便之后丢给别的软件处理。

### US4：不混淆语言概念

作为用户，我在右侧“应用/界面”设置里选择中文/English/Deutsch；同时在会议/转写设置里看到只读“转写语言：中文”，知道当前不是 UI 语言选择，也不会误以为 App 在翻译或识别英文。

## 5. 功能需求

### FR1 三栏中控台布局

- 保留左侧 `NavigationSplitView` 会议历史列表。
- detail 区改为横向中控台：中间主工作区 + 右侧设置检查器。
- 中间工作区应在视觉上占主要宽度；右侧设置应有固定/最大宽度。
- 空状态应提示“从左侧选择会议或开始新会议”。

### FR2 中间主工作区

中间必须包含：

- 当前会议标题/状态。
- 录音状态（未录音、录音中、停止/完成、本地转写中、转写完成/失败）。
- 主按钮：开始录音、停止、转写选中会议、取消转写。
- 延迟实时字幕/最终片段预览：复用 `rollingPreview`。
- 会议详情与导出入口：打开文件夹、打开机器 Markdown/SRT/JSON、打开用户编辑 Markdown（若存在）。
- 录音中强调字幕区域；转写完成后强调 Markdown 编辑区域。

### FR3 轻量 Markdown 编辑

- 新增用户编辑版 Markdown 文件，建议路径：`<meeting>/transcript/edited.md`。
- `final.md` 是机器生成原版，绝不因用户编辑而覆盖。
- 选中会议时：
  - 若 `edited.md` 存在，编辑器加载 `edited.md`。
  - 否则若 `final.md` 存在，编辑器加载 `final.md` 作为初始内容，但保存时写入 `edited.md`。
  - 否则显示空状态/提示等待本地转写。
- 提供保存按钮：保存到 `edited.md`。
- 若用户修改后切换会议、刷新历史或转写完成触发重新加载，系统应自动保存当前会议的未保存编辑到 `edited.md`，避免会议中弹窗或数据丢失。
- 保存必须绑定编辑内容来源会议，不能把 A 会议编辑内容误写入 B 会议。
- 可提供“从机器原文重新载入”按钮，但必须明确会丢弃当前编辑器未保存内容；第一版可以只做保存，避免破坏性操作。
- `exportArtifacts` 需要包含 Edited Markdown/用户编辑版 Markdown，并正确标记是否存在。

### FR4 语言概念分离

- UI language picker 移出录音设置，放到右侧“应用设置/界面”组，并使用明确标签“界面语言”。
- 转写语言第一版固定中文：显示只读 `转写语言：中文`。
- 不新增会议语言 picker，不新增翻译选项。
- 继续保持 Whisper 配置为中文识别、不翻译英文。

### FR5 右侧设置检查器

建议分组：

1. 常用
   - 输出目录。
   - 会议标题。
   - 转写语言：中文（只读）。
2. 本地 Whisper
   - 当前模型。
   - whisper.cpp 可执行文件。
   - 重新扫描/手动选择。
3. 实时草稿
   - 滚动延迟。
   - 分块大小。
4. 应用
   - 界面语言。
5. 权限/诊断（高级折叠）
   - 麦克风、系统音频、输出文件夹、沙盒/权限、模型状态。

录音中右侧设置应自动弱化：例如折叠高级组、减小视觉权重、禁用会造成混乱的修改项，中心字幕优先。

### FR6 本地优先和统一 ScreenCaptureKit 路径保持不变

- 不改变现有 `NativeMacCaptureProvider`/统一录音路径。
- 不引入云端 API。
- 不新增网络依赖。

## 6. 验收标准

1. 详情区不再是单列堆叠；用户能明显看到“左历史 / 中会议 / 右设置”的结构。
2. UI 语言选择显示为“界面语言”，且不在录音/会议语言区域。
3. 会议/转写语言第一版显示为只读“转写语言：中文”。
4. 录音中中间区域显示录音状态和延迟实时字幕草稿。
5. 停止录音并完成本地转写后，中间可看到最终转写内容。
6. 用户可在纯文本 Markdown 编辑器中修改内容并保存。
7. 保存用户编辑内容会生成 `transcript/edited.md`，不覆盖 `transcript/final.md`。
8. 若已有 `edited.md`，重新选择该会议时优先加载用户编辑版。
9. 切换会议或转写完成前，已修改但未手动保存的 Markdown 会自动保存到原会议 `edited.md`。
10. 导出/打开入口显示机器 Markdown、SRT、JSON，以及用户编辑版 Markdown 的存在状态。
11. 现有 Swift tests 通过；新增 AppModel 测试覆盖 edited Markdown、autosave、防串写行为。
12. 打包后的 `dist/VoiceCaptioner.app` 默认中文 UI，并包含本地模型/whisper 可执行文件现有能力。

## 7. 文案原则

- 中文默认。
- 对实时功能使用“延迟实时草稿/延迟实时转写”，避免“实时转写”暗示零延迟。
- 对 Markdown 使用“机器原文”和“用户编辑版”区分，降低误删/覆盖风险。
- 语言文案使用“界面语言”和“转写语言”，禁止只写“语言”。
